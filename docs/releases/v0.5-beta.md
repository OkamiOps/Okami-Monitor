# OKAMI Mission Control 0.5-beta

Beta release focused on turning OKAMI Mission Control into a simpler multi-agent operations dashboard for technical and non-technical users.

> Package version: `0.5.0-beta`
> GitHub tag: `v0.5-beta`

## Executive Summary

`0.5-beta` consolidates the dashboard around a clearer operating model: one **Agents** screen for server SSH, connected runtimes, dashboard access and Okami API Keys. The product language is now English-first, with PT-BR still available from the language selector, and the main UI copy no longer frames the monitor as a Hermes-only cockpit.

This release includes:

- unified agent setup;
- automatic Okami API Key creation per agent;
- a simpler onboarding path for users with little technical experience;
- corrected token metrics for `1h`, `24h`, `7d` and `30d`;
- broader real data collection over SSH;
- Kanban board aggregation and richer task details;
- hardened API Key and secret storage;
- English-first README, changelog and release notes with Portuguese sections.

## Major Features

### 1. Unified Agents Screen

The previous Config/Agents/Hermes split has been consolidated into **Agents**. Users now have one place to:

- prepare dashboard access;
- configure the SSH connection for the agent server;
- connect default and external agent runtimes;
- inspect agent files and instances;
- run safe diagnostics;
- see permissions, pending setup and connection status;
- open advanced key tools only when support or backup is needed.

This removes the ambiguity of choosing between separate configuration pages and makes the intended workflow easier to understand.

### 2. Automatic Okami API Key per Agent

The old manual key flow is no longer the default path. Users should not need to generate a key, copy it, open a remote file and paste it by hand.

When **Connect agent** is clicked, the backend:

1. Normalizes the runtime registration.
2. Creates an Okami API Key with the recommended scopes.
3. Stores the secret in the encrypted server vault.
4. Registers only prefix, fingerprint and a safe secret reference on the runtime.
5. Attempts to write `.okami.env` to the remote workspace over SSH.

If SSH is not configured yet, the key remains prepared in the vault and can be applied after the server connection is completed.

### 3. English-First Product Language

The dashboard now defaults to English. PT-BR remains available through the language switcher and the selected language is persisted in `localStorage`.

The release covers:

- navigation;
- section headings;
- status cards;
- buttons and forms;
- empty states;
- Usage recommendations;
- Kanban states;
- Agents setup;
- Apps, Docs, CLIs, Logs, Skills, Sessions, Office and Pixel copy.

The main overview heading changed from a Hermes-specific label to **Operation in real time** so the product can support multiple runtimes without implying that every workflow belongs to Hermes.

### 4. Token Metrics Fixed

The Overview token cards now use one coherent aggregation source.

Fixes include:

- the main **Tokens** card follows the selected period;
- `1h`, `24h`, `7d` and `30d` no longer compare incompatible buckets;
- empty days are represented as zero in daily windows;
- the side period card and the main token card stay aligned;
- demo token data varies coherently instead of looking fixed.

This resolves the impossible cases where 7 or 30 days could appear smaller than 24 hours.

### 5. Kanban and Real Agent Data

Kanban now handles multiple boards and richer backend records.

Included improvements:

- detected board aggregation;
- board selector;
- stable selection when state refreshes;
- real task cards when backend records are available;
- task detail with status, result, description, skills, comments, work log and run history;
- clearer empty states when auxiliary Kanban tables have no data.

### 6. Office and Pixel Stability

The Office and Pixel Office flows now align agents with canonical agent state more consistently.

The stray `agentops-triage-4` pixel entity was removed from the visible office when it does not belong to the canonical agent list. Pixel Office also preserves canvas and layout state more carefully during data refreshes, reducing flicker and unnecessary remounts.

### 7. Backend, Security and Persistence

The backend now supports a more realistic agent dashboard flow:

- generated Okami API Keys with scoped access;
- SHA-256 key hash validation;
- AES-256-GCM encrypted secrets;
- restricted local storage permissions;
- auth status, bootstrap and key-management endpoints;
- SSH config and SSH test endpoints;
- safe command execution;
- allowed remote file writes;
- agent runtime registration and connection;
- same-origin production proxy support;
- loopback development access to reduce setup friction;
- remote `.okami.env` writes with `chmod 600`.

### 8. Documentation

Documentation was rewritten to match the new product shape:

- README is now English-first with a Portuguese version section.
- CHANGELOG is now English-first with a Portuguese summary.
- This release note is English-first with a Portuguese section.
- The docs explain Quick Start, scripts, architecture, API Keys, scopes, Agents, SSH, security and validation.
- The docs clarify that some internal collector and endpoint names still use legacy Hermes naming while the product surface is multi-agent.

## Technical Highlights

- `src/App.jsx`: UI copy cleanup, English default language, unified Agents flow, translation coverage and token/Kanban rendering updates.
- `src/data/mockMissionControl.js`: demo copy moved away from Hermes-only labels and refreshed for multi-agent operations.
- `src/lib/apiClient.js`: API client coverage for auth, API Keys, runtimes, SSH and commands.
- `src/lib/useMissionControl.js`: same-origin SSE and polling fallback resilience.
- `server/index.js`: auth, scopes, protected endpoints, runtime connect and SSH actions.
- `server/store.js`: local storage, encrypted secrets and API Key hashes.
- `server/hermesCollector.js`: real SSH collection for sessions, Kanban, logs, skills, config and registry data.
- `src/PixelOfficeCanvas.jsx`: canvas lifecycle stability.
- `src/styles.css`: styles for Agents, access states, filters, panels and operational controls.

## Breaking or Behavioral Changes

- **Agents** is the primary visible setup screen.
- `#config` remains supported internally for old links.
- Manual API Key generation is advanced/support functionality, not the normal user path.
- English is the default UI language.
- Production deployments should prefer same-origin `/api` unless a direct API URL is intentionally configured.
- The public release is `v0.5-beta`; the package version remains `0.5.0-beta` for SemVer compatibility.

## How to Test

```bash
npm install
cp .env.example .env.local
npm run dev:all
```

Open `http://localhost:5173`.

Recommended checks:

1. Switch between EN and PT-BR in the top bar.
2. Open Overview and switch `1h`, `24h`, `7d` and `30d`.
3. Open **Agents**.
4. Test the local/demo flow without manually pasting a token.
5. Register an external agent.
6. Click **Connect agent**.
7. Confirm the UI does not ask the user to manually edit a file for the key.
8. Open Kanban, Office, Pixel, Sessions, Logs, Skills, Apps, CLIs and Docs.
