# 🚀 OKAMI Mission Control `0.5-beta`

> A cleaner, English-first, multi-agent operations dashboard for OKAMI.

`0.5-beta` is the first beta focused on making Mission Control understandable for both technical and non-technical users. The main workflow is now centered on one place: **Agents**. From there, users can configure the agent server over SSH, connect runtimes, create Okami API Keys automatically and inspect operational files without manually editing secrets.

## 🔑 Automatic Okami API Keys

The old flow was too manual. This release moves the complexity to the backend.

When the user clicks **Connect agent**, the API now:

1. Normalizes the runtime registration.
2. Creates an Okami API Key with recommended scopes.
3. Stores the full secret in the encrypted server vault.
4. Registers only safe metadata on the runtime.
5. Tries to write `.okami.env` to the remote workspace over SSH.

```env
OKAMI_API_KEY=okami_key_...
OKAMI_API_BASE_URL=https://your-dashboard.example.com/api
OKAMI_AGENT_ID=main-agent
OKAMI_AGENT_NAME=Main Agent
```

If SSH is not configured yet, the key stays prepared in the vault and can be applied after the server connection is completed.

### Key Safety Model

| Data | Stored as | Notes |
| --- | --- | --- |
| Full API Key | Encrypted secret | Only available at creation/application time |
| Token hash | SHA-256 hash | Used for validation |
| Token prefix | Plain metadata | Safe for UI listing |
| Scopes | Plain metadata | Required for access decisions |
| SSH/private secrets | AES-256-GCM encrypted | Stored in server-side vault |

## 📊 Token Metrics Fixed

The Overview token cards now use one coherent aggregation source.

### Fixed behavior

- The main **Tokens** card follows the selected period.
- `1h`, `24h`, `7d` and `30d` use compatible buckets.
- Empty days are represented as zero in daily windows.
- The side period card and the main token card stay aligned.
- Demo token data varies coherently instead of appearing fixed.

```text
1h   → last hour buckets
24h  → last 24 hourly buckets
7d   → last 7 daily buckets
30d  → last 30 daily buckets
```

This resolves the regression where `7d` or `30d` could show fewer tokens than `24h`.
